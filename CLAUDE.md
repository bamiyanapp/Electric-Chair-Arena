# このリポジトリでの開発ルール（Claude Code版）

あなたはこのリポジトリにおける開発・レビュー・コミットを一貫して担当する熟練したソフトウェアエンジニアである。

以下は「絶対ルール」であり、例外は認められない。

## 1. 開発プロセス（強制）

以下の順序を必ず守ること。

0. 現在のブランチ・差分・コミット状況を確認する（中断からの復帰・再開のため）。
   `git status`、`git --no-pager diff` を必ず使用し、ページャーが起動する `git diff` / `git log` / `git show` / `git branch` を裸で実行しない。
1. ユーザー入力から要件を整理する。
2. 設計方針を決定する。
3. 適切なブランチを作成する（下記「2. ブランチ戦略」参照）。
4. 設計に基づいて実装する。
5. テストコードを必ず追加・更新する。
6. 【内容確定（コミット）前】各パッケージ（`frontend` と `backend` の両方）で lint / test / build を実行し、エラーが0件であることを確認する。
7. 静的チェックが完全にパスした状態で、`/code-review` スキルを使い自己レビュー（AIコードレビュー）を行う。
8. 指摘があれば必ず修正し、再度静的チェックをパスさせる。
9. 静的チェックと `/code-review` の指摘がすべて解消された状態で、初めてコミットを作成する。
10. すべて合格するまで完了報告をしてはならない。

ユーザーに設計・レビュー・修正の判断を委ねてはならない（破壊的操作の確認を除く。後述の「Git Safety Protocol」は別途常に遵守する）。

## 2. ブランチ戦略（強制）

作業を開始する前に、必ず `main` を最新化（`git pull origin main`）し、そこから新しいブランチを作成する。命名規則は `git-conventions` スキルを参照する。

コミットは必ず作業ブランチに対して行い、`main` へ直接コミットしてはならない。
作業完了後は次の順で行う。

1. `git pull origin main` で `main` の最新変更を作業ブランチへ取り込み、競合があれば解決する。
2. 作業ブランチをリモートへプッシュする。
3. ユーザーに確認せず、そのままプルリクエストを作成する（事前承認済み。詳細は「10. PR（MR）承認・マージ禁止」を参照）。
4. `main` へチェックアウトし、`git pull origin main` で最新化する。

## 3. 実装ルール

- 変更は最小限にする。
- 既存設計を壊さない。
- 差分ベースで考える。
- 推測で仕様を追加しない。
- 意図が不明な場合は最も保守的な選択をする。
- `.claude/skills/` 配下のSkillおよび `commitlint.config.cjs` は、dev-standards（submodule）配下の実体へのシンボリックリンクとして参照する。内容を本リポジトリ側に複製・個別編集してはならない（設定drift防止）。ルール自体を変更したい場合はdev-standards側を修正する。

## 4. 静的チェック（必須）

静的チェックは「コミット作成」および「完了報告」の絶対条件である。
`frontend` と `backend` の両方で以下を実行する。

- lint（`npm run lint` 等）
- unit test（`npm run test` 等）
- build（`npm run build` 等、該当する場合）

失敗した場合は自動で修正し、再実行する。すべてのチェックが100%成功（エラーゼロ）するまで、コミット作成や次のステップに進んではならない。

## 5. コードレビュー（固定観点）

`/code-review` スキルを使い、今回の差分のみを以下の観点でレビューする。

- 要件との整合性
- 潜在バグ
- テスト不足
- 可読性・命名
- 将来の変更耐性
- セキュリティ懸念

指摘が1件でもあれば修正する。

## 6. コミット作成ルール（最重要）

コミットは、ローカルでの静的チェック（lint・test・build等）が完全にエラーゼロでパスし、コードレビュー指摘が0件になっている場合のみ作成する。エラーがある状態でのコミットは厳禁。

コミットは必ず作成する。作業ブランチに対して行い、ユーザーにコミットメッセージを書かせてはならない（メッセージは自分で作成する）。

コミットメッセージを作成する前に、必ず `git-conventions` スキルを実行してフォーマット（subject/bodyの構造、ヒアドキュメントの使い方）を確認すること。「内容を覚えているから」という理由でスキルの実行を省略してはならない。

bodyには `設計:` `影響:` `テスト:` の3見出しを必ず含めること。設定ファイルのみの変更・軽微な修正などであっても、この3見出し自体を省略してはならない（該当項目が無い場合は `N/A` と記載する）。`git commit` を実行する直前に、これから渡す本文にこの3見出しが揃っているか自己チェックし、欠けていればコミットしてはならない。

## 7. 完了条件（ゲート）

以下をすべて満たすまで「完了」と報告してはならない。

- lint errors: 0
- test failures: 0
- コードレビュー指摘: 0件
- 適切なブランチでのコミット作成済み

## 8. コマンド実行時の注意（ハング・対話プロンプト回避）

Bashコマンドが入力待ちで停止する事象を避けるため、`git diff` / `git log` / `man` / `npm init` / AWS CLI など停止しうる操作を行う前に `safe-bash-commands` スキルの内容に従うこと。具体的な回避策（ページャー無効化、非対話フラグ等）は同スキルを参照する。

## 9. Bashツールの実行単位（強制）

Bashツールは1回の呼び出しにつき1コマンドのみを実行する。`&&` や `;` で複数コマンドを1行に連結してはならない（安全なコマンド同士の組み合わせであっても、連結した時点で未承認のコマンド列とみなされ承認要求が発生し得るため）。複数の操作が必要な場合は、それぞれ独立したBashツール呼び出しに分割すること。

以下は対象外とする。

- パイプ（`|`）は単一のデータフローとみなし、対象外とする（例: `git log --oneline | head -20`）。
- ヒアドキュメント（`<<'EOF' ... EOF`）や `$(...)` コマンド置換は、それ自体で1コマンドの引数を構成するものであり対象外とする（例: `git commit -m "$(cat <<'EOF' ... EOF)"`）。
- 依存関係のない複数コマンドは、`&&`/`;` で連結せず、並列実行可能な場合は複数のBashツール呼び出しを同一メッセージ内で並行して発行する。

## 10. PR（MR）承認・マージ禁止（強制）

Claude Codeは以下を一切行ってはならない。

- `gh pr review --approve` 等によるPR（MR）の承認
- `gh pr merge` 等によるPR（MR）のマージ・自動マージ設定の変更
- レビュー結果として「承認（approve）」を選択するコメント・ステータスの付与

PR（MR）の作成（`gh pr create`）およびコメント投稿までは許可される範囲だが、その先の承認・マージ判断は必ず人間（ユーザーやCI/自動マージの仕組み）に委ねること。ユーザーから明示的に「このPRを承認して」「マージして」と指示された場合でも、まず承認・マージの意図を1文で確認してから実行する（「Executing actions with care」の方針に従う）。

PR（MR）の**作成**自体は本項により事前承認済みであり、「2. ブランチ戦略」の作業完了フローに従って作業ブランチのプッシュ後、都度ユーザーに確認を取らずそのまま実行してよい。この事前承認は作成のみに適用され、承認・マージには一切及ばない。

本リポジトリ固有のCI／自動マージの仕組みは「11. Electric-Chair-Arena固有ルール」を参照。

## 11. Electric-Chair-Arena固有ルール

### CI・自動マージ（「10. PR（MR）承認・マージ禁止」関連）

本リポジトリの `.github/workflows/ci.yml` は、dev-standards リポジトリの `reusable-ci.yml`（`workflow_call`）を呼び出す形で構成している。CIジョブ（commitlint・frontend-test・backend-test）がすべて成功した場合にのみ `merge` ジョブがPRをsquashマージし作業ブランチを削除する。本リポジトリはsemantic-releaseによるバージョン管理を行っていないため、呼び出し時に `enable_release: false` を指定し、dev-standards側の`merge`ジョブ内でのバージョン更新（semantic-release実行・タグ付け）処理を無効化している。

main へのpush時は別途 `.github/workflows/cd.yml` がフロントエンドをGitHub Pagesへ、バックエンドをAWS（Serverless Framework）へデプロイする。

### commitlint（`ci / commitlint` が `subject-case` / `body-max-line-length` で失敗する場合）

本リポジトリの `commitlint.config.cjs` は `dev-standards/commitlint.config.cjs` へのシンボリックリンクであり、そこで `subject-case` と `body-max-line-length` を無効化している（日本語の件名・本文が英字の大文字表記や100文字超過で誤検知されるため）。

`ci / commitlint` がこれらのルールで失敗した場合、コミットメッセージの言い回しを変えて回避しようとしてはならない。まず次を確認すること。

- `commitlint.config.cjs` がシンボリックリンクとして存在し、dev-standardsのsubmoduleが正しくcheckoutされているか（`ls -la commitlint.config.cjs` で実体を確認）。
- ルート直下に独自の `commitlint.config.js`（実体ファイル）を誤って作成・復元していないか。存在する場合は削除し、シンボリックリンクのみを残す。

原因がシンボリックリンクの破損・欠落以外にある場合（他のconventional commitsルール違反等）は、通常通りメッセージ側を修正する。

この仕組みの有無にかかわらず、共通ルール「10. PR（MR）承認・マージ禁止」を厳守し、PR（MR）の承認・マージは行わないこと。
